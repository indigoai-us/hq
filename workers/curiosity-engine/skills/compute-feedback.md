# compute-feedback

Measure knowledge delta for the current cycle, write a cycle report, update cumulative metrics, and recalibrate observation weights for self-improvement.

## Arguments

`$ARGUMENTS` = optional cycle date override (YYYY-MM-DD). Defaults to today.

## Inputs

- **Queue:** `workspace/curiosity/queue.yaml`
- **Knowledge tree:** `knowledge/knowledge-tree.yaml`
- **Observations:** `workspace/curiosity/observations/*.yaml`
- **Cumulative metrics:** `workspace/curiosity/metrics.yaml`
- **Previous cycle reports:** `workspace/curiosity/cycles/*.yaml`

## Process

### 1. Determine cycle date

Use `$ARGUMENTS` if provided, otherwise use today's date in YYYY-MM-DD format.

### 2. Count observations processed

Read all files in `workspace/curiosity/observations/`. Count the total number of observation files. Group them by `type` field (knowledge_gap, stale_revalidation, coverage_gap, pattern_gap).

### 3. Analyze the queue

Read `workspace/curiosity/queue.yaml`. For each question, note:

- **questions_generated:** Total count of all questions in the queue
- **questions_resolved:** Count where `status: resolved`
- **questions_wont_fix:** Count where `status: wont_fix`
- **questions_open:** Count where `status: open`
- **questions_in_progress:** Count where `status: in_progress`

Group resolved and wont_fix questions by their `type` field (from the question schema) to compute per-type resolution rates.

### 4. Count knowledge entries

Read `knowledge/knowledge-tree.yaml`. Count total entries across all domains by summing `entry_count` at the domain level. This is the **current** knowledge entry count.

If a previous cycle report exists (check `workspace/curiosity/cycles/` for the most recent YAML file before today), read its `coverage_after` value as the **before** count. Otherwise, use the current count as both before and after (first cycle).

Compute:
- **knowledge_entries_created:** current - before (if positive, else 0)
- **knowledge_entries_updated:** This is harder to track precisely. For the simple implementation, set to 0 and note it as a future enhancement.

### 5. Compute confidence changes

Scan the knowledge tree for entries that have a `confidence` field. Compare against the previous cycle report's `confidence_snapshot` (if it exists). Record any entries where confidence changed:

```yaml
confidence_changes:
  - file: "knowledge/path/to/file.md"
    before: 0.8
    after: 0.9
```

If this is the first cycle, record all entries with confidence as the initial snapshot (before = null, after = current value).

### 6. Identify top domains

From the knowledge tree, rank domains by `entry_count` descending. Take the top 5 as `top_domains`.

Also identify the most active domains this cycle by looking at which domains had questions resolved.

### 7. Write cycle report

Write the report to `workspace/curiosity/cycles/{YYYY-MM-DD}.yaml`:

```yaml
# Curiosity Engine — Cycle Report
# Generated by: curiosity-engine worker (compute-feedback skill)

cycle_date: "YYYY-MM-DD"
generated_at: "ISO 8601 timestamp"

observations_processed: N
questions_generated: N
questions_resolved: N
questions_wont_fix: N
questions_open: N

knowledge_entries_created: N
knowledge_entries_updated: 0

coverage_before: N  # knowledge tree entry count at start
coverage_after: N   # knowledge tree entry count now
coverage_delta: N   # after - before

top_domains:
  - domain: "domain-name"
    entry_count: N

observation_type_breakdown:
  knowledge_gap: N
  stale_revalidation: N
  coverage_gap: N
  pattern_gap: N

resolution_by_type:
  knowledge_gap:
    resolved: N
    wont_fix: N
    total: N
    wont_fix_rate: 0.0
  stale_revalidation:
    resolved: N
    wont_fix: N
    total: N
    wont_fix_rate: 0.0
  coverage_gap:
    resolved: N
    wont_fix: N
    total: N
    wont_fix_rate: 0.0
  pattern_gap:
    resolved: N
    wont_fix: N
    total: N
    wont_fix_rate: 0.0

confidence_changes:
  - file: "path"
    before: 0.8
    after: 0.9

confidence_snapshot:
  - file: "path"
    confidence: 0.9
```

### 8. Update cumulative metrics

Read `workspace/curiosity/metrics.yaml`. Update:

- **total_cycles:** increment by 1
- **total_questions_resolved:** add this cycle's resolved count
- **avg_questions_per_cycle:** total_questions_resolved / total_cycles
- **knowledge_growth_rate:** compute as total coverage delta / total_cycles (use cycle reports history)
- **top_gap_domains:** re-rank from queue — domains with the most open questions
- **observation_type_distribution:** add this cycle's observation counts to the running totals

### 9. Sensor recalibration

This is the self-improving mechanism. For each observation type:

1. Look at the `resolution_by_type` data computed in step 3
2. If a type has at least 3 total questions (to avoid small-sample noise) AND its `wont_fix_rate` exceeds 50%, it is producing low-value questions
3. Reduce its weight in `observation_weights` by multiplying by 0.8 (minimum floor: 0.2)
4. Log the recalibration to `recalibration_log` with date, type, old weight, new weight, and reason

Conversely, if a type has a wont_fix_rate below 20% and its weight is below 1.0, increase weight by multiplying by 1.1 (maximum ceiling: 1.0). This allows recovery.

Write the updated metrics file.

### 10. Report results

Print a summary to stdout:

```
Cycle {date} feedback complete:
- Observations: {N} processed
- Questions: {generated} total, {resolved} resolved, {wont_fix} wont_fix, {open} open
- Knowledge: {created} new entries, coverage {before} -> {after} (delta: {delta})
- Confidence changes: {N} entries changed
- Recalibrations: {list any weight changes, or "none"}
```

## Output

- Cycle report at `workspace/curiosity/cycles/{YYYY-MM-DD}.yaml`
- Updated `workspace/curiosity/metrics.yaml`

## Rules

- Never delete or modify queue entries — this skill is read-only on the queue
- Never modify knowledge files — this skill only reads the knowledge tree
- If metrics.yaml is missing or corrupt, recreate it with zeroed values
- All timestamps in ISO 8601 format
- Cycle reports are immutable once written — if re-running the same date, overwrite (idempotent)
- The recalibration weights in metrics.yaml are consumed by the scan-and-prioritize skill when scoring question priority. This creates the feedback loop: low-value observation types get deprioritized automatically.
