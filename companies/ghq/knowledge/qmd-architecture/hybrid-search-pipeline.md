QMD's `query` command implements a three-stage hybrid search pipeline: BM25 keyword search, vector semantic search, and LLM-based cross-encoder reranking. Results from BM25 and vector stages are fused using Reciprocal Rank Fusion (RRF) before being passed to the reranker.

The pipeline begins with **query expansion**: the user's query is expanded by an LLM into multiple sub-queries typed as `lex` (keyword), `vec` (semantic), or `hyde` (hypothetical document embedding). Each sub-query is routed to its appropriate backend. An optional `intent` parameter (added in v1.1.5) steers all five pipeline stages — expansion, strong-signal bypass, chunk selection, reranking, and snippet extraction — to disambiguate queries like "performance" without searching on the intent string itself.

BM25 search runs against a SQLite FTS5 index for fast keyword matching. Vector search uses embeddings generated by a local GGUF model via node-llama-cpp, stored in the same SQLite database. The reranker (Qwen3-Reranker GGUF) scores candidate results with cross-encoder attention. Models stay loaded in VRAM across requests when using the MCP HTTP server, with embedding/reranking contexts disposed after 5 min idle and recreated on demand.

## RRF Implementation

QMD uses **Reciprocal Rank Fusion** to merge up to 6 ranked lists (original + 2 expanded queries × BM25 + vector each):

```
score(doc) = Σ [ weight(list) × 1 / (k + rank(doc, list)) ]
```

Key parameters (rank is 0-indexed):

| Parameter | Value | Purpose |
|-----------|-------|---------|
| **k** | **60** | Fusion smoothing constant |
| Original query weight | **2.0** | User's actual words are a stronger signal |
| Expanded query weight | **1.0** | Lower weight for LLM-generated variants |

**Top-rank bonus** — prevents exact matches from being diluted by expansion results:
- Rank #1 in any list: **+0.05**
- Ranks #2–3 in any list: **+0.02**

## Position-Aware Blending (RRF → Reranker)

After LLM reranking, final scores blend RRF and reranker signals based on candidate position:

| RRF Rank | RRF Weight | Reranker Weight | Rationale |
|----------|-----------|-----------------|-----------|
| 1–3      | 75%       | 25%             | Preserve high-confidence retrieval matches |
| 4–10     | 60%       | 40%             | Balanced blend |
| 11+      | 40%       | 60%             | Trust reranker for lower candidates |

This prevents the reranker from destroying strong keyword matches while allowing semantic relevance to surface lower-ranked results.

## Strong-Signal Bypass

When BM25 returns a clear match, LLM expansion and reranking are skipped. Both conditions must hold:

1. Top result score ≥ **0.85** (`STRONG_SIGNAL_MIN_SCORE`)
2. Score gap between rank 1 and rank 2 ≥ **0.15** (`STRONG_SIGNAL_MIN_GAP`)

The bypass is **automatically disabled** when an `intent` parameter is provided — ambiguous queries may need expansion even if the initial BM25 match looks strong.
