## HQ Registry (Worker Packs)

Worker packs, command sets, and knowledge bases are published to the HQ registry at `https://admin.getindigo.ai` using the `hq` CLI.

### Prerequisites

- `hq` CLI installed (`npm i -g @{company}ai/hq-cli`)
- An account at `admin.getindigo.ai`

### Workflow

```bash
# 1. Authenticate
hq login

# 2. Scaffold (for new packs)
hq create-worker
# Follow prompts — generates hq-package.yaml, worker.yaml, skills/, README.md

# 3. Edit hq-package.yaml — set name, version, description, exposes
# 4. Validate without uploading
hq publish --dry-run

# 5. Publish
hq publish
```

### hq-package.yaml fields

| Field | Required | Description |
|-------|----------|-------------|
| `name` | ✓ | Unique registry name (slug) |
| `type` | ✓ | `worker-pack`, `command-set`, `skill-bundle`, `knowledge-base`, `company-template` |
| `version` | ✓ | Semver (e.g. `1.0.0`) |
| `description` | ✓ | Short description shown in search results |
| `exposes.workers` | — | Paths to worker.yaml files (relative to package root) |
| `exposes.commands` | — | Paths to command files |
| `exposes.skills` | — | Paths to skill files |
| `requires.packages` | — | Other HQ packages that must be installed first |
| `hooks.on-install` | — | Shell script run after install |
| `hooks.on-remove` | — | Shell script run before removal |

### Trust model

Publishers must be explicitly trusted by the installer before `hq install` will proceed without a warning:

```bash
hq trust add your-org
```

The trusted publishers list is stored at `packages/trusted-publishers.json`.

## npm Organization

All packages publish under the `@{company}ai` npm scope (except `create-hq` which is unscoped for `npx` convenience).

| Package | npm Name | Scoped |
|---------|----------|--------|
| packages/create-hq | create-hq | No |
| packages/hq-cli | @{company}ai/hq-cli | Yes |
| packages/hq-cloud | @{company}ai/hq-cloud | Yes |

## Prerequisites

- npm account with publish access to `@{company}ai` org
- `NPM_TOKEN` secret set in GitHub repo settings
- Two-factor authentication enabled on npm account

## Manual Publishing

### 1. Bump version

All packages use unified versioning. Update version in all three package.json files:

```bash
# packages/create-hq/package.json
# packages/hq-cli/package.json
# packages/hq-cloud/package.json
# Also update root package.json
```

### 2. Build

```bash
npm run build
```

### 3. Publish (order matters)

```bash
cd packages/hq-cloud && npm publish --access public
cd packages/hq-cli && npm publish --access public
cd packages/create-hq && npm publish --access public
```

hq-cloud first (hq-cli depends on it), then hq-cli, then create-hq.

## Automated Publishing (CI/CD)

The `publish.yml` workflow triggers on version tags:

```yaml
# Triggered by: git tag v5.0.0 && git push --tags
on:
  push:
    tags: ['v*']
```

Workflow steps:

1. Checkout code
2. Setup Node.js 18 with npm registry
3. `npm ci` — install all deps
4. `npm run build` — build all packages
5. `npm run typecheck` — verify types
6. Publish hq-cloud → hq-cli → create-hq (in order)

### Creating a release

```bash
# After merging to main:
git tag v5.1.0
git push --tags
```

This triggers the publish workflow automatically.

## Template Bundling

The `create-hq` package includes the `template/` directory in its published bundle. The package.json `files` field controls what's included:

```json
{
  "files": ["dist", "template"]
}
```

When users run `npx create-hq`, npm downloads the package including the template, and scaffold.ts copies it to their chosen directory.

### Updating the template

1. Make changes in `template/` directory
2. Test locally: `node packages/create-hq/dist/index.js ~/test-hq`
3. Bump version and publish

## Version Strategy

- **Major**: Breaking changes to CLI interface or template structure
- **Minor**: New features, new commands, new workers
- **Patch**: Bug fixes, content updates, documentation

All packages share the same version number for simplicity. A single `git tag` triggers publishing of all three packages.
