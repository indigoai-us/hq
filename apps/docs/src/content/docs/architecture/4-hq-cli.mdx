The `@indigoai/hq-cli` package is the ongoing management CLI for HQ. It handles four concerns: **package management** (install, remove, update, search, publish worker packs to the HQ registry), **kernel governance** (protecting and verifying core files via `core.yaml`), **module management** (syncing external knowledge repos), and **cloud sync** (bridging to `@indigoai/hq-cloud` for S3-backed mobile access).

Built with [Commander.js](https://github.com/tj/commander.js), the CLI registers command groups:

```
hq install|remove|update|list|search   # Package management
hq login|publish|trust                 # Registry auth + publishing
hq create-worker                       # Worker pack scaffolding
hq doctor                              # HQ health diagnostics
hq core status                         # Kernel integrity check
hq modules <add|sync|list|update>      # Module management
hq sync <init|start|stop|status|push|pull>  # Cloud sync
```

## Package Management

The package ecosystem lets users install pre-built worker packs, command sets, and knowledge bases from the HQ registry at `https://admin.getindigo.ai`.

### Package format: `hq-package.yaml`

Every installable package includes an `hq-package.yaml` manifest:

```yaml
name: dev-team
type: worker-pack          # worker-pack | command-set | skill-bundle | knowledge-base | company-template
version: 1.0.0
minHQVersion: "6.5.0"
description: "Full dev team worker pack"
author: getindigo
exposes:
  workers:
    - workers/architect/worker.yaml
    - workers/backend-dev/worker.yaml
  commands: []
  skills: []
  knowledge: []
requires:
  packages: []
  services: []
hooks:
  on-install: scripts/on-install.sh
  on-remove: scripts/on-remove.sh
```

Package types:
- `worker-pack` — one or more worker.yaml files + skills
- `command-set` — slash commands for `.claude/commands/`
- `skill-bundle` — skill files for `.claude/skills/`
- `knowledge-base` — markdown knowledge repos
- `company-template` — full company scaffold

### `hq install <package>`

Resolves dependencies (topological sort via Kahn's algorithm), fetches tarballs from the registry or falls back to `git clone` if the registry is unreachable, validates SHA256 checksums, extracts to the HQ tree, runs `on-install` hooks, and records the installation in `packages/installed.json`.

```bash
hq install dev-team
hq install dev-team --company acme   # company-scoped install
```

The `--company` flag installs workers into `companies/{slug}/workers/` instead of the shared `workers/` tree.

**Offline fallback:** If the registry returns a non-2xx response, `install` falls back to `git clone` using the `repo` field from the package manifest.

### `hq remove <package>`

Reads `packages/installed.json`, removes all tracked files, runs `on-remove` hooks if present, then removes the package entry. Warns if dependent packages still require it.

```bash
hq remove dev-team
```

### `hq update [package]`

Re-fetches the latest version from the registry, removes old files, installs new files, and updates `packages/installed.json`. If no package name is given, all installed packages are updated.

```bash
hq update dev-team   # Update one package
hq update            # Update all
```

### `hq list`

Reads `packages/installed.json` and displays a formatted table of installed packages with name, version, type, install date, and scope.

```bash
hq list   # or: hq packages list
```

### `hq search <query>`

Queries `GET /api/packages?q={query}` on the registry and displays matching packages with name, type, version, description, and download count.

```bash
hq search dev-team
hq search "worker pack"
```

## Registry Auth & Publishing

### `hq login`

Authenticates with the HQ registry. Stores the bearer token at `~/.hq/auth.json`.

```bash
hq login   # Prompts for credentials
```

### `hq publish`

Packages the current directory as an HQ package and uploads to the registry. Requires `hq-package.yaml` at the current directory root. Creates a tarball, posts to `/api/packages` (new) or `/api/packages/{name}/versions` (update), then cleans up.

```bash
hq publish           # Publish or update from cwd
hq publish --dry-run # Validate without uploading
```

### `hq trust`

Manages the trusted publishers list at `packages/trusted-publishers.json`. The install command warns (but does not block) when installing from an untrusted publisher.

```bash
hq trust add getindigo
hq trust remove getindigo
hq trust list
```

## Worker Pack Scaffolding

### `hq create-worker`

Interactive wizard that scaffolds a publishable worker pack directory with the correct structure for the HQ registry:

```
my-worker/
├── hq-package.yaml
├── worker.yaml
├── skills/
│   └── example-skill.md
└── README.md
```

```bash
hq create-worker        # Interactive prompts
hq create-worker my-worker --type worker-pack
```

## Diagnostics

### `hq doctor`

Runs 5 health checks and outputs a formatted pass/warn/fail table. Exits with code 1 if any check fails.

| Check | What it verifies |
|-------|-----------------|
| Required binaries | `claude` and `gh` are on PATH |
| Package integrity | Every file recorded in `installed.json` exists on disk |
| Symlink health | No broken symlinks in `workers/` or `knowledge/` |
| Core integrity | Locked files in `core.yaml` match stored checksums |
| Registry reachable | `GET /api/packages` returns 2xx |

```bash
hq doctor
```

### `hq core status`

Reads `core.yaml` at the HQ root, computes SHA256 of each locked file (or directory tree), and compares against stored checksums. Displays a per-file pass/fail table. Exits with code 1 if any locked file has been modified.

```bash
hq core status
```

`core.yaml` classifies files into three tiers:
- `locked` — kernel files (CLAUDE.md, settings.json, hooks/) — checksummed, block on edit
- `reviewable` — editable with warning (workspace/, commands/, skills/)
- `open` — unrestricted (`**`)

The `protect-core.sh` PreToolUse hook enforces the locked tier at edit time by computing a live checksum and comparing against `core.yaml`.

## Registry Client

`src/utils/registry-client.ts` provides a `RegistryClient` class used by all package commands:

| Method | Endpoint | Description |
|--------|----------|-------------|
| `listPackages(query?)` | `GET /api/packages` | List/search packages |
| `getPackage(name)` | `GET /api/packages/{name}` | Fetch package metadata |
| `getDownloadInfo(name)` | `GET /api/packages/{name}/download` | Get presigned download URL |
| `publishPackage(formData)` | `POST /api/packages` | Publish new package |
| `publishVersion(name, formData)` | `PUT /api/packages/{name}/versions` | Publish new version |
| `downloadTarball(url, dest, checksum?)` | — | Download + SHA256 validate |

The client reads auth from `~/.hq/auth.json`, retries on network errors and 5xx responses (max 2 retries), enforces a 30s timeout for API calls and 120s for downloads, and uses `HQ_REGISTRY_URL` env var to override the default `https://admin.getindigo.ai`.

## Dependency Resolution

`src/utils/dep-resolver.ts` implements topological sort via Kahn's algorithm. When installing a package that declares `requires.packages`, the resolver builds a dependency graph and produces an install order that ensures dependencies are installed first. Circular dependencies are detected and reported as an error.

## Module Commands

### `hq modules add <repo-url>`

Registers a module in `modules.yaml`. Validates the repo URL, parses the repo name, builds a `ModuleDefinition`, and appends it to the manifest.

```bash
hq modules add https://github.com/org/knowledge-ralph.git \
  --as ralph \
  --branch main \
  --strategy link \
  --path "docs:knowledge/public/Ralph"
```

| Option | Default | Description |
|--------|---------|-------------|
| `--as <name>` | Parsed from URL | Module name |
| `--branch <branch>` | `main` | Git branch to track |
| `--strategy <strategy>` | `link` | Sync strategy: `link`, `merge`, or `copy` |
| `--path <src:dest>` | `.:workers/<name>` | Path mapping (repeatable) |

### `hq modules sync`

The core sync loop. For each module in the manifest:

1. **Clone or fetch** the module repo into `modules/<name>/`
2. **Apply the sync strategy** (link, merge, or copy) based on path mappings
3. **Record the commit SHA** in `modules.lock`

```bash
hq modules sync          # Sync all to latest
hq modules sync --locked # Sync to locked commits
```

The `--locked` flag checks out the exact commit recorded in `modules.lock` instead of pulling latest. This enables reproducible installs across machines.

On first run, `modules/` is automatically added to `.gitignore` since cloned repos should not be tracked by HQ git.

### `hq modules list`

Displays all modules with their status, current commit, lock state, and whether they are behind the remote.

```bash
hq modules list   # or: hq modules ls
```

### `hq modules update [module-name]`

Fetches the latest from the remote, pulls, and updates `modules.lock` with the new commit SHA. If no module name is given, all modules are updated.

```bash
hq modules update ralph   # Update one module's lock
hq modules update         # Update all locks
```

## Type System

All types are defined in `src/types.ts`:

```typescript
type SyncStrategy = 'link' | 'merge' | 'copy';
type AccessLevel = 'public' | 'team' | `role:${string}`;

interface PathMapping {
  src: string;   // Path within module repo
  dest: string;  // Path in HQ tree (relative to HQ root)
}

interface ModuleDefinition {
  name: string;
  repo: string;              // Git URL (https or git@)
  branch?: string;           // Default: main
  strategy: SyncStrategy;
  paths: PathMapping[];
  access?: AccessLevel;      // For future RBAC
}

interface ModulesManifest {
  version: '1';
  modules: ModuleDefinition[];
}

interface ModuleLock {
  version: '1';
  locked: Record<string, string>; // module name -> commit SHA
}
```

The manifest stores modules as an array (not a record), so ordering is preserved. The lock file maps module names to commit SHAs for deterministic installs.

### SyncState

The merge strategy tracks per-file state for conflict detection:

```typescript
interface SyncState {
  version: '1';
  files: Record<string, {
    hash: string;          // SHA256 of file content at sync time
    syncedAt: string;      // ISO timestamp
    fromModule: string;    // Module that provided this file
  }>;
}
```

This is persisted to `.hq-sync-state.json` at the HQ root and used by the merge strategy to detect local modifications.

## Sync Strategies

Each strategy is implemented as an async function that takes a `ModuleDefinition`, the module's local directory, and the HQ root, returning a `SyncResult`.

### Link Strategy

**File:** `src/strategies/link.ts`

Creates relative symlinks from the module repo into the HQ tree. This is the default strategy and is used for knowledge bases where files should remain in their git repo and be transparently accessible from HQ paths.

For each path mapping:
1. Validates the source path exists in the module repo
2. Creates parent directories at the destination if needed
3. If an existing symlink is found at the destination, removes and recreates it
4. If a real file or directory exists, warns and skips (does not overwrite)
5. Computes a **relative** symlink path for portability across machines

Relative symlinks mean the link still works if the entire HQ directory is moved, as long as the internal structure is preserved.

### Merge Strategy

**File:** `src/strategies/merge.ts`

Copies files from the module repo into the HQ tree with SHA256-based conflict detection. This is used for content that should be owned by HQ after initial sync (commands, templates, configuration).

The merge logic for each file:
1. Compute the SHA256 hash of the source file
2. If the destination file exists, compute its hash
3. Look up the hash recorded at last sync time from `.hq-sync-state.json`
4. **Skip (conflict)** if the destination was modified since last sync *and* differs from the source — the user's local changes are preserved
5. **Skip (unchanged)** if source and destination hashes match
6. **Copy** the file and record the new hash in sync state

This three-way comparison (source, destination, last-synced) prevents overwriting local customizations while still pulling upstream changes for unmodified files.

### Copy Strategy

Planned. Currently falls through to the merge strategy implementation.

## Cloud Commands

**File:** `src/commands/cloud.ts`

The `hq sync` command group bridges to `@indigoai/hq-cloud` via dynamic imports. Each subcommand lazily loads the cloud package and delegates to its exported functions:

| Command | Cloud Function | Description |
|---------|---------------|-------------|
| `hq sync init` | `initSync()` | Authenticate with {Product}AI and provision S3 storage |
| `hq sync start` | `startDaemon()` | Start the background file-watching sync daemon |
| `hq sync stop` | `stopDaemon()` | Stop the daemon |
| `hq sync status` | `getStatus()` | Display sync health: running state, last sync time, file count, bucket, errors |
| `hq sync push` | `pushAll()` | Force upload all local changes to S3 |
| `hq sync pull` | `pullAll()` | Force download all cloud changes to local |

Dynamic imports (`await import("@indigoai/hq-cloud")`) mean the cloud package is only loaded when cloud commands are actually invoked. Module management commands work without `hq-cloud` installed.

## Data Files

The CLI reads and writes three data files at the HQ root:

| File | Format | Purpose |
|------|--------|---------|
| `modules.yaml` | YAML | Module manifest — the source of truth for what modules exist |
| `modules.lock` | YAML | Locked commit SHAs for reproducible installs |
| `.hq-sync-state.json` | JSON | Per-file hash tracking for merge conflict detection |
